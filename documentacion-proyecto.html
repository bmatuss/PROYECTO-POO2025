Sistema POS para Restaurante

Documentación Técnica del Proyecto

1. Descripción del Problema

En muchos restaurantes, la toma de pedidos y el control de mesas se realizan de manera manual lo que genera errores, demoras y una gestión ineficiente. Este proyecto propone el desarrollo de un sistema interactivo de gestión de pedidos y mesas, implementado en C++ con Qt, que optimice la atención al cliente y mejore la organización interna del restaurante.

2. Análisis del Problema

Elementos y Entes Participantes

  • Actores Externos: Cliente, Garzón, Cocinero, Cajero
  • Entidades del Sistema: Mesa, Pedido, Item (Comida/Bebida), Usuario
  • Procesos Principales: Toma de pedidos, Preparación en cocina, Procesamiento de pagos

Definición del Sistema

Sistema POS (Point of Sale) para restaurantes que automatiza la gestión de pedidos, desde la toma de comandas hasta la preparación en cocina y bar.

3. Definición de Requerimientos

Casos de Uso Principales

CU-001: Tomar un Pedido

Actor Principal: Garzón/Mesero

Nivel: Objetivo del Usuario

Precondiciones:

  • El sistema POS está iniciado
  • Las categorías y menú están cargados
  • Hay mesas disponibles
  • El garzón tiene acceso al sistema

Descripción: Este caso de uso describe el proceso de tomar un pedido desde que un cliente se sienta en una mesa hasta que la orden se envía a cocina y/o bar para su preparación.

Flujo Principal:

1. Selección de Mesa
  1. El Garzón abre la aplicación POS (PosWindow)
  2. El sistema muestra todas las mesas disponibles con botones dinámicos
  3. El Garzón selecciona la mesa del cliente (ejecuta mesaSeleccionada())
  4. El sistema carga el estado de la mesa y crea un nuevo Pedido si es necesario
2. Navegación del Menú
  1. El sistema muestra las categorías del menú disponibles (botones creados por crearBotonesCategorias())
  2. El Garzón selecciona una categoría (ejecuta categoriaSeleccionada())
  3. El sistema actualiza categoriaActual y muestra los items de esa categoría
  4. El sistema ejecuta crearBotonesItems() para mostrar los productos disponibles
3. Agregar Items al Pedido
  1. El Garzón selecciona un item del menú (ejecuta itemSeleccionado())
  2. El sistema busca el item usando buscarItemEnCategoria(nombreItem)
  3. El Garzón especifica la cantidad y comentarios especiales si es necesario
  4. El sistema crea un PedidoItem(item, cantidad, comentario)
  5. El sistema agrega el item al pedido actual usando pedidoActual->agregarItem()
  6. Repetir pasos 6-13 para cada item adicional
4. Envío del Pedido
  1. El Garzón presiona el botón "Enviar" (ejecuta on_enviarButton_clicked())
  2. El sistema separa el pedido por tipo:
    • Items tipo "Comida" → se envían a cocinawindow
    • Items tipo "Bebida" → se envían a barrawindow
  3. El sistema agrega el pedido a la mesa usando mesa->agregarPedido(pedido)
  4. El sistema actualiza las listas de pedidos en las ventanas correspondientes

Postcondiciones:

  • Éxito:
    • El pedido se ha registrado en el sistema
    • Los items de comida aparecen en la ventana de cocina
    • Los items de bebida aparecen en la ventana de bar
    • La mesa tiene el pedido asociado

Clases Principales: PosWindow, Mesa, Pedido, PedidoItem, Item, Comida, Bebida, Categoria

Frecuencia de Uso: Alta - Utilizado múltiples veces por hora durante el servicio del restaurante.

CU-002: Preparar Orden en Cocina

Actor Principal: Cocinero

Nivel: Objetivo del Usuario

Precondiciones:

  • El sistema de cocina está activo (cocinawindow)
  • Se ha recibido al menos una orden de comida desde el POS
  • Los ingredientes y equipo de cocina están disponibles

Descripción: Este caso de uso describe el proceso de gestión y preparación de órdenes de comida en la cocina, desde la recepción de la orden hasta marcar el plato como listo para servir.

Flujo Principal:

1. Recepción de Orden
  1. El Cocinero abre la ventana de cocina (cocinawindow)
  2. El sistema muestra las órdenes pendientes en la interfaz
  3. Una nueva orden llega desde el POS (ejecuta método de recepción de pedido)
  4. El sistema actualiza automáticamente la lista de órdenes pendientes
  5. El sistema muestra los detalles de la orden:
    • Número de mesa
    • Items de comida solicitados
    • Cantidades de cada item
    • Comentarios especiales (ej: "sin mayo", "extra picante")
2. Preparación de la Comida
  1. El Cocinero lee los detalles específicos de cada PedidoItem de tipo "Comida"
  2. El Cocinero revisa los comentarios especiales para cada item
  3. El Cocinero inicia la preparación física de los platos
3. Finalización y Marcado como Listo
  1. El Cocinero completa la preparación de todos los items de comida de la orden
  2. El Cocinero verifica que todos los platos estén correctos según las especificaciones
  3. El Cocinero marca listo
  4. El sistema remueve de la lista de pendientes

Postcondiciones:

  • Éxito:
    • La orden de comida está completamente preparada
    • Todos los items están marcados como "listos"
    • La orden se remueve de la lista de pendientes

Clases Principales: cocinawindow, Pedido, PedidoItem, Comida, Item

Frecuencia de Uso: Alta - Utilizado continuamente durante las horas de servicio del restaurante.

CU-003: Procesar Pago de Mesa

Actor Principal: Cajero/UsuarioCaja

Actor Secundario: Garzón

Nivel: Objetivo del Usuario

Precondiciones:

  • La mesa tiene al menos un pedido registrado
  • Los pedidos de la mesa están completos (comida y bebidas servidas)
  • El cliente solicita la cuenta
  • El sistema POS está operativo

Descripción: Este caso de uso describe el proceso de calcular, presentar y procesar el pago total de una mesa, incluyendo todos los pedidos realizados durante la visita del cliente.

Flujo Principal:

1. Solicitud de Cuenta
  1. El Cliente solicita la cuenta al Garzón
  2. El Garzón accede al sistema POS (PosWindow)
  3. El Garzón selecciona la mesa correspondiente usando buscarMesaPorNumero()
  4. El Garzón presiona el botón "Pagar" (ejecuta on_pagarButton_clicked())
2. Cálculo del Total
  1. El sistema obtiene todos los pedidos de la mesa usando mesa->getPedidos()
  2. Para cada Pedido en la mesa:
    • El sistema calcula el monto usando pedido->getMontoTotal()
    • Se suman todos los subtotales de cada PedidoItem
  3. El sistema calcula el total final usando mesa->getTotalMesa()
3. Procesamiento del Pago
  1. El Cajero ejecuta procesarPago() del UsuarioCaja
  2. El sistema registra el pago y actualiza total_recaudado
  3. Se actualiza el total del cajero usando setTotalRecaudado()
4. Finalización y Cierre de Mesa
  1. El sistema confirma el pago exitoso
  2. El sistema ejecuta mesa->cerrarMesa() para cambiar estado a CERRADA
  3. La mesa queda disponible para nuevos clientes

Postcondiciones:

  • Éxito:
    • El pago se registra exitosamente en el sistema
    • El total del cajero se actualiza correctamente
    • La mesa cambia a estado CERRADA
    • La mesa está disponible para nuevos clientes

Clases Principales: PosWindow, Mesa, Pedido, PedidoItem, UsuarioCaja, Item

Frecuencia de Uso: Alta - Utilizado al final de cada servicio de mesa durante todo el día de operación.

4. Diseño

Diagrama de Clases UML

Explicación del diagrama de clases: El diagrama muestra la arquitectura completa del sistema POS con las siguientes características principales:
  • Herencia: La clase Item es la clase base de la cual heredan Comida y Bebida, implementando polimorfismo para el manejo diferenciado de productos
  • Composición: La clase Mesa contiene una colección de Pedido, y cada Pedido contiene múltiples PedidoItem
  • Agregación: PedidoItem mantiene una referencia a Item pero no controla su ciclo de vida
  • Interfaces de Usuario: Las clases PosWindow, CocinaWindow y BarraWindow representan las diferentes interfaces del sistema
  • Gestión de Usuarios: Jerarquía de usuarios con Usuario como clase base y especializaciones para UsuarioGarzon y UsuarioCaja
  • Organización del Menú: La clase Categoria organiza los items del menú, mientras que MenuLoader se encarga de la carga de datos

Dificultades Encontradas y Soluciones

Durante el desarrollo del sistema POS se presentaron diversas dificultades técnicas que requirieron soluciones creativas:

  • Comunicación entre Ventanas: Una de las principales dificultades fue implementar el envío de objetos (pedidos) entre las diferentes ventanas del sistema (POS, Cocina, Bar). Inicialmente intentamos implementar el patrón Publisher-Subscriber utilizado en tareas anteriores del semestre, pero resultó ser demasiado complejo para este contexto específico. - Solución: Después de investigar las capacidades de Qt, descubrimos el sistema de signals y slots nativo del framework (usando `emit` y `connect()`). Esta solución resultó ser mucho más elegante y apropiada para la comunicación entre componentes de Qt, permitiendo una comunicación asíncrona y desacoplada entre ventanas.
  • Gestión de Estados de Mesa: El manejo del estado de las mesas y la sincronización entre pedidos activos presentó complejidad adicional. - Solución: Se implementó un sistema de estados básico en la clase Mesa que controla las transiciones entre estados ABIERTA y CERRADA

Funcionalidades Faltantes

Aunque el sistema cumple con los objetivos principales, quedaron algunas funcionalidades pendientes por implementar:

  • Gestión Completa de Usuarios: El sistema tiene implementadas las clases UsuarioGarzon y UsuarioCaja, pero no se integró completamente el control de acceso y permisos por rol. Falta implementar login, sesiones de usuario y restricciones de funcionalidades según el tipo de usuario.
  • Persistencia de Datos: Los pedidos y transacciones no se guardan , por lo que se pierden al cerrar la aplicación.
  • Configuración Dinámica: El número de mesas (16) y la configuración del menú están hardcodeados en lugar de ser configurables.
  • Sistema de Impresión: Falta la funcionalidad para imprimir tickets de venta y comandas para cocina/bar.

Bugs Conocidos

Se han identificado los siguientes problemas menores que requieren atención:

  • Navegación de Mesa: Una vez que se selecciona una mesa en la interfaz principal, no existe un botón "Volver" o "Cancelar" para regresar al menú de selección de mesas sin tener que enviar un pedido o procesar un pago. Aunque no es técnicamente un bug, es una funcionalidad de UX que falta y puede resultar confusa para el usuario.
  • Validación de Pedidos Vacíos: El sistema permite enviar pedidos sin items, lo que puede generar comandas vacías en cocina y bar.
  • Comentarios en Pedidos: No existe interfaz gráfica para agregar comentarios especiales a los items del pedido. Los comentarios solo pueden agregarse modificando directamente el código fuente (hardcodeado), lo que limita la funcionalidad en tiempo real del sistema.
  • Manejo de Errores de Carga: Si el archivo menu.json no se encuentra o tiene errores de formato, el sistema no maneja correctamente el error y puede fallar al iniciar.
